# 01. 대시보드 워크플로우: 팀 프로젝트 안내서

## 같이 분석하고, 같이 만든다

팀 프로젝트를 시작하기 전에 읽어야 할 안내서예요.

**처음부터 읽기** — 섹션 1~4: 협업 원칙, 전체 흐름, 단계별 작업 가이드  
**필요할 때 찾아보기** — 섹션 5~8: 파일 관리, 문서 협업, 시트 취합, 트러블슈팅  
**한 페이지 요약** — 섹션 9: 원칙 정리

## 1. 프로젝트 개요

Python과 태블로를 활용해 서울시 대중교통 데이터(버스/지하철/자전거 중 선택)를 분석하고 대시보드를 만들어요.

### 팀 구성

- **팀원 수**: 3명 이상
- **도구**: Python (Google Colab) + 태블로 퍼블릭 데스크톱 앱
- **협업**: Google Drive + Zep (화면 공유) + 디스코드 (소통)
- **결과물**: 태블로 퍼블릭에 배포된 대시보드 + 분석 노트북

### 프로젝트 목표

| 단계 | 목표 |
|------|------|
| 질문 정의 | 팀이 답하고 싶은 비즈니스 질문을 함께 결정 |
| Python 분석 | 팀 전체가 동일하게 EDA · 전처리 · 인사이트 도출 |
| 태블로 시각화 | 담당 시트를 나눠 제작 후 대시보드 조립 |
| 배포 및 발표 | 태블로 퍼블릭 업로드 후 팀 발표 |

## 2. 협업 원칙

### Python은 함께, 태블로는 나눠서

역할을 너무 일찍, 너무 완전히 나누면 학습 기회를 잃어요.

```
피해야 할 패턴:
  팀원 A → Python 전담
  팀원 B → 태블로 전담
  결과:   A는 시각화를 못 해보고, B는 분석을 못 해봐요
```

이 프로젝트에서는 아래 원칙을 지켜요.

| 단계 | 방식 | 이유 |
|------|------|------|
| 질문 정의 | 팀 전체 함께 | 방향이 다르면 나중에 다시 해야 해요 |
| Python EDA · 전처리 | 팀 전체 함께 | 모든 팀원이 데이터를 이해해야 해요 |
| 태블로 시트 제작 | 분담 또는 화면 공유 | 동시 편집이 안 되는 도구 특성상 나눌 수밖에 없어요 |
| 대시보드 조립 · 발표 | 팀 전체 함께 | 최종 결과물은 팀 모두가 이해하고 설명할 수 있어야 해요 |

### "각자 하고 나중에 모이자"가 안 되는 이유

```
피해야 할 패턴:
  "우리 전처리 각자 진행하고 한두 시간 뒤에 모여서 얘기하죠"
```

전처리는 코딩이 아니라 **판단의 연속** 이에요.
이상치를 제거할까 말까, 결측치를 0으로 채울까 버릴까, 파생 변수를 어떻게 정의할까.
이런 결정을 각자 내리면, 두 시간 뒤에 세 가지 버전의 데이터가 나와요.
어느 것을 쓸지 다시 논의해야 하고, 탈락된 작업은 버려져요.

흩어졌다 모이는 방식은 시간을 절약하는 것처럼 보이지만, 합치는 비용이 더 커요.
그래서 이 프로젝트에서는 **한 명이 코딩하고 나머지가 보면서 토론** 하는 방식을 사용해요.

데이터를 직접 다뤄본 사람과 그렇지 않은 사람은 발표 때 차이가 바로 드러나요.

> "이 이상치를 왜 제거했나요?"  
> "이 파생 변수는 어떻게 계산했나요?"

이 질문에 팀 전체가 답할 수 있어야 해요.
데이터를 직접 다뤄본 경험이 없으면 대답하기 어려워요.

### 의견이 갈릴 때 의사결정 규칙

```
다수결로 결정 → 동수가 나오면 해당 단계 담당자가 결정 → 그래도 결론이 안 나면 팀장이 최종 결정
```

- 킥오프 단계 담당자 = 팀장
- Python EDA 담당자 = 코딩 담당자
- 태블로 시트 제작 담당자 = 조작 담당자
- 완벽한 결정보다 빠른 결정이 중요해요

## 3. 전체 워크플로우

```
[원본 데이터]
     │
     ▼
[ 1단계: 질문 정의 ]  ── 팀 전체, 킥오프 미팅
     │                   비즈니스 질문 3개 이하 합의
     │                   데이터 · 독자 · 차트 구성 계획
     │
     ▼
[ 2단계: Python EDA · 전처리 ]  ── 팀 전체, 코랩 공유
     │                              한 명이 코딩, 나머지가 함께 보며 토론
     │                              분포 확인, 이상치 처리, 파생 변수 생성
     │                              정제 파일 저장 → Drive에 자동 반영
     │
     ▼
[ 3단계: 태블로 시트 제작 ]  ── 담당 시트 분담 또는 화면 공유
     │                          각자 담당 차트 시트 완성
     │
     ▼
[ 4단계: 대시보드 조립 ]  ── 한 명이 조립, 전원 Zep 화면 공유로 참여
     │                       시트 취합, 필터·액션 연결, 서식 통일
     │
     ▼
[ 5단계: 배포 및 발표 ]  ── 팀 전체
                            태블로 퍼블릭 업로드 + 팀 발표
```

> Python 단계가 끝나기 전에 태블로 작업을 시작하면 안 돼요.  
> 데이터가 바뀌면 태블로에서 만든 차트를 처음부터 다시 확인해야 해요.

### 1단계: 질문 정의 (팀 전체, 킥오프 미팅)

> **킥오프 전 — 팀장이 디스코드에 미리 공지해요**
>
> ```
> 안녕하세요! 킥오프 미팅 전에 아래 내용을 준비해 오세요.
>
> 1. 분석할 교통수단(버스/지하철/자전거) 생각해두기
>    (서울 열린데이터 광장 data.seoul.go.kr 에서 어떤 데이터가 있는지 둘러봐요)
> 2. 미팅 링크: [Zep 링크]
> 3. 미팅 일시: YYYY-MM-DD HH:MM
> ```
>
> 사전 공지가 없으면 킥오프에서 갑자기 교통수단을 정해야 해서 시간이 오래 걸려요.

#### 진행 순서 (40~50분)

**1. 공유 폴더 및 협업 문서 생성 (10분)**

팀장이 진행해요.  
팀장은 이후 킥오프 단계에서 의견이 동수로 갈릴 때 최종 결정권을 가져요.  
나머지 팀원은 화면 공유로 보면서 공유 링크를 받아요.

```
[Google Drive 폴더 생성]
1) drive.google.com 접속
2) [+ 새로 만들기] → [폴더] → 폴더명 입력  예: transit_team
3) 폴더 우클릭 → [공유] → 팀원 이메일 입력 → [편집자] 권한 → [공유]
4) 공유된 폴더 링크를 디스코드 채널에 붙여넣기

[하위 폴더 구조 생성]
폴더 안에서 [+ 새로 만들기] → [폴더]로 아래 5개 폴더를 만들어요.
  data/raw/   data/clean/   notebooks/   tableau/   docs/
  (각 폴더의 용도는 섹션 5(파일 관리) 참고)

[협업 문서 생성]
폴더 안에서 아래 두 파일을 만들어요 (섹션 6(문서 협업) 참고)
  - [+ 새로 만들기] → [Google 문서]        이름: 킥오프회의록_YYYYMMDD
  - [+ 새로 만들기] → [Google 스프레드시트]  이름: transit_collab
```

> 이후 회의 내용은 Google Docs 회의록에 실시간으로 기록하면서 진행해요.  
> 한 명이 기록 담당을 맡고, 나머지가 말하는 방식이 효율적이에요.

**2. 교통수단 결정 (10분)**

```
진행 방법:
  → 각자 생각해온 교통수단(버스/지하철/자전거)을 돌아가며 소개
  → 간단한 투표(디스코드 이모지 반응 등)로 결정
  → 동수가 나오면 팀장이 최종 결정해요 (의사결정 규칙은 섹션 2(협업 원칙) 참고)

데이터 출처:
  → 서울 열린데이터 광장 (data.seoul.go.kr)
  → 서울특별시 교통 빅데이터 플랫폼 (t-data.seoul.go.kr)
  → 공공데이터포털 (data.go.kr) 에서 "서울 버스/지하철/자전거" 검색

추천 데이터셋:
  지하철: "서울교통공사 역별 일별 시간대별 승하차인원" (서울 열린데이터 광장)
  버스:   "서울시 버스노선별 정류장별 승하차인원" (서울 열린데이터 광장)
  자전거: "서울특별시 공공자전거 이용정보(일별)" (공공데이터포털)
  → CSV 형식 다운로드를 권장해요
  → 엑셀(.xlsx)은 코랩에서 추가 라이브러리(openpyxl)가 필요해요

#### 의견 충돌 시 처리 방법

#### 필수 작업 코드

```python
# 0. Google Drive 마운트 (코랩 첫 셀에서 실행)
from google.colab import drive
drive.mount('/content/drive')

# 프로젝트 경로 설정 (본인 공유 폴더 경로에 맞게 수정)
PROJECT = '/content/drive/MyDrive/transit_team'

import pandas as pd

# 1. 데이터 불러오기 & 기본 탐색
df = pd.read_csv(f'{PROJECT}/data/raw/서울시_지하철_이용현황.csv')
df.info()          # 컬럼 타입 확인
df.isnull().sum()  # 결측치 확인
df.describe()      # 기초 통계 확인

# 2. 이상치 탐지 (IQR 방식)
Q1 = df['승차인원'].quantile(0.25)
Q3 = df['승차인원'].quantile(0.75)
IQR = Q3 - Q1
outliers = df[(df['승차인원'] < Q1 - 1.5 * IQR) | (df['승차인원'] > Q3 + 1.5 * IQR)]
print(f'이상치 {len(outliers)}건 발견')
# 제거 여부는 팀이 함께 결정하고, Google Sheets 탭 2에 근거를 기록해요

# 3. 결측치 처리
df = df.dropna(subset=['역명'])  # 또는 fillna()

# 4. 파생 변수 생성
df['총이용객'] = df['승차인원'] + df['하차인원']
df['Year'] = pd.to_datetime(df['날짜']).dt.year
df['Month'] = pd.to_datetime(df['날짜']).dt.month

# 5. 정제 파일 저장 (Google Drive에 직접 저장)
df.to_csv(f'{PROJECT}/data/clean/clean_20260305.csv', index=False, encoding='utf-8-sig')
print('저장 완료:', df.shape)
```

4. 시트 배치 (Z-패턴: KPI → 주요 차트 → 보조 차트)
5. 필터 연결: 필터 우클릭 → [워크시트에 적용] → [모든 항목]
6. 액션 설정: [대시보드] → [작업] → 클릭 필터 추가
7. 서식 통일: 폰트·색상·테두리 점검
8. [파일] → [패키지 통합 문서로 저장]으로 최종본 저장
```

**조립 중 팀원 소통 방식**

```
조립 담당자가 배치하면서 팀원에게 물어요:
  "KPI 카드를 왼쪽 위에 두는 게 나을까요, 오른쪽 위?"
  "필터를 상단에 둘까요, 사이드에 둘까요?"

팀원은 채팅 또는 마이크로 바로 의견:
  "왼쪽 위가 자연스러워요" / "사이드가 공간 절약돼요"

수정 요청은 구체적으로:
  ✗ "좀 더 예쁘게 해주세요"
  ✓ "이용객 차트 시트 제목 폰트를 14pt로 맞춰주세요"
  ✓ "라인 차트 Y축 0에서 시작하도록 고정해주세요"

버전 저장 규칙:
  → 의견 반영할 때마다 버전 올려 저장
     dashboard_v1_YYYYMMDD.twbx → dashboard_v2_YYYYMMDD.twbx
  → Drive에 업로드하고 디스코드에 "v2 올렸어요" 알림
```

### 5단계: 배포 및 발표

#### 태블로 퍼블릭 업로드

```
1. 완성된 대시보드 파일을 데스크톱 앱에서 열어요
2. [파일] → [태블로 퍼블릭에 저장] 선택
3. 태블로 퍼블릭 계정으로 로그인 후 업로드
4. 업로드 완료 후 자동으로 브라우저가 열리며 공개 링크 확인
5. 링크를 복사해서 디스코드에 공유
```

#### 태블로 퍼블릭 작품 페이지 꾸미기 (포트폴리오용)

업로드 직후 브라우저에서 작품 페이지를 꾸며요.  
포트폴리오로 공유할 때 제목·설명이 없으면 보는 사람이 무슨 분석인지 알 수 없어요.

```
브라우저에서 대시보드 페이지 열린 상태에서:

1. 제목 설정
   → 대시보드 제목 클릭 후 수정
   예: "서울시 지하철 이용 현황 분석 대시보드 (2025~2026)"
   → 검색되기 쉬운 키워드 포함: 교통수단, 분석 주제, 연도

2. 설명(Description) 작성
   → [편집] → [설명] 항목에 3~5줄 작성
   예:
     "서울시 지하철 이용 현황 데이터를 기반으로 호선별·시간대별 이용객을
      분석한 팀 프로젝트입니다.
      Python으로 EDA·전처리 후 태블로로 시각화했어요.
      주요 인사이트: 2호선 이용객 1위, 출퇴근 시간대 혼잡도 집중"

3. 썸네일 확인
   → 자동 생성된 썸네일이 대시보드를 잘 보여주는지 확인
   → 필요하면 대시보드 레이아웃을 조정 후 재업로드

4. 링크 최종 공유
   → 팀원 전체에게 공개 URL 공유
```

#### 개인 포트폴리오에 저장하기

팀 프로젝트는 한 명의 계정에 업로드되어 있어요.  
나머지 팀원도 **본인 이름으로 포트폴리오를 남기려면** 각자 계정에 동일한 파일을 업로드해요.

```
[각 팀원이 개인 계정에 업로드하는 방법]
1. Drive에서 최종 대시보드 파일(dashboard_vN_YYYYMMDD.twbx) 다운로드
2. 태블로 퍼블릭 데스크톱 앱에서 파일 열기
3. [파일] → [태블로 퍼블릭에 저장] → 본인 계정으로 로그인 후 업로드
4. 작품 페이지 제목·설명을 동일하게 작성
   → 제목 끝에 "(팀 프로젝트)" 또는 담당 파트를 추가해도 좋아요

[태블로 퍼블릭 프로필 관리]
→ 프로필 URL: public.tableau.com/app/profile/본인아이디
→ 프로필 사진·소개글도 채워두면 포트폴리오로서 완성도가 높아져요
→ 작품이 쌓일수록 포트폴리오 페이지가 풍부해져요
```

> 팀 프로젝트를 개인 계정에 중복 업로드하는 것은 괜찮아요.  
> 단, 본인이 기여한 부분을 설명에 명시해요. 예: "EDA·전처리 및 호선별 이용객 차트 담당"

#### 발표 준비 (총 10~15분, 질문 포함)

**시간 배분**

팀원 수에 맞게 파트를 나눠요. 아래를 참고해요.

| 파트 | 시간 |
|------|------|
| 데이터 소개 | 2분 |
| 분석 과정 (EDA·전처리 근거) | 3분 |
| 대시보드 시연 + 인사이트 | 5분 |
| 결론 및 제언 | 2분 |
| 질문 답변 | 3분 |

> 팀원이 많으면 시연 파트를 차트별로 나눠서 더 많은 팀원이 발표해요.  
> 시연 파트는 한 명이 클릭·조작하고, 각 차트 설명은 담당자가 말해요.

**발표 스크립트 작성 방법**

Google Docs 발표 스크립트 문서에 위 시간 배분표를 참고해서 파트별로 담당자를 배정해요.  
스크립트 템플릿은 **섹션 6(문서 협업)** 에 있어요.

**예상 질문 준비**

발표 전에 팀 전체가 아래 질문에 답할 수 있는지 확인해요.

```
데이터 관련:
  Q. 이 데이터의 출처는 어디인가요? 신뢰할 수 있나요?
  Q. 분석 기간을 왜 이 기간으로 설정했나요?

전처리 관련:
  Q. 이상치를 왜 제거했나요? 제거하지 않으면 결과가 달라지나요?
  Q. 파생 변수는 어떻게 계산했나요?

차트 관련:
  Q. 이 차트에서 가장 중요한 포인트는 무엇인가요?
  Q. 다른 차트 유형을 쓰지 않은 이유가 있나요?

결론 관련:
  Q. 제언을 실제로 적용하면 어떤 효과를 기대할 수 있나요?
  Q. 이 분석에서 한계점은 무엇인가요?
```

> 모든 질문에 완벽하게 답하지 않아도 돼요.  
> "저희가 분석한 범위에서는 이러한 결과였어요. 더 깊은 분석은 추가로 필요해요."도 좋은 답변이에요.

## 5. 파일 관리

### Google Drive 폴더 구조

팀 공유 폴더를 아래 구조로 만들어요.  
코랩 노트북은 Drive 안에서 직접 생성·편집하므로 별도 업로드가 필요 없어요.

```
transit_team/              ← Google Drive 공유 폴더
├── data/
│   ├── raw/                  ← 원본 데이터 (절대 수정 금지)
│   └── clean/                ← 코랩에서 정제한 데이터 (태블로가 읽는 파일)
├── notebooks/                ← 코랩 노트북 (.ipynb)
│   └── analysis_YYYYMMDD.ipynb
├── tableau/
│   ├── sheets/               ← 팀원별 시트 .twbx 파일
│   └── dashboard/            ← 최종 조립 .twbx 파일
└── docs/                     ← Google Docs · Sheets 내보내기 파일
                                 (회의록·발표 스크립트 PDF 등)
```

### 파일 공유 방법

- 코랩 노트북은 Drive 공유 폴더 안에 만들면 팀원이 **링크로 바로 접속** 가능
- 태블로 파일(.twbx)은 Drive에 업로드 → 팀원이 다운로드해서 로컬에서 열기
- 파일명에 날짜 또는 버전 포함: `dashboard_v1_20260303.twbx`
- Drive의 [버전 기록]으로 이전 파일 복원 가능

### 파일명 규칙

```
# 정제 데이터
clean_YYYYMMDD.csv              예: clean_20260305.csv

# 코랩 노트북
analysis_YYYYMMDD.ipynb         예: analysis_20260303.ipynb

# 태블로 시트 파일 (분담 방법 B 사용 시)
sheet_차트이름_담당자.twbx       예: sheet_monthly_trend_B.twbx

# 태블로 최종 대시보드
dashboard_v버전_YYYYMMDD.twbx   예: dashboard_v1_20260303.twbx

# Google Docs 문서
문서종류_YYYYMMDD               예: 킥오프회의록_20260303, 발표스크립트_20260311

# Google Sheets 협업 문서
프로젝트명_collab         예: transit_collab
```

> **주의**: 같은 `.twbx` 파일을 두 명이 동시에 열고 저장하면 한 명의 작업이 사라져요.  
> 작업 전에 디스코드에 "파일명 작업 시작" 메시지를 남겨서 충돌을 방지해요.

### Google Docs — 문서 협업

```

### Google Sheets — 표 협업

컬럼 정의, 전처리 결정 기록, 차트 담당 배분처럼 **표 형태로 관리할 정보** 에 사용해요.

#### 시트 구성

**시트 1: 컬럼 정의서**

데이터의 각 컬럼이 무엇을 의미하는지 팀 전체가 공통으로 이해할 수 있도록 기록해요.

| 컬럼명 | 한국어 이름 | 타입 | 설명 | 비고 |
|--------|------------|------|------|------|
| 날짜 | 날짜 | 날짜 | 이용 날짜 | |
| 역명 | 역명 | 문자 | 지하철 역 이름 | |
| 승차인원 | 승차인원 | 숫자 | 해당 역의 승차 인원 수 | |

## 7. 태블로 시트 취합 방법

방법 B(분담)를 선택한 경우, 팀원이 만든 시트를 하나의 파일로 모아야 해요.

> **태블로 퍼블릭 데스크톱 앱은 파일 간 시트 복사·붙여넣기를 지원하지 않아요.**
> 유료 Tableau Desktop에서만 가능한 기능이에요.
> 대신 아래 방법으로 시트를 취합해요.

### 취합 방법: 조립 담당자가 시트를 직접 재현

팀원이 만든 시트를 조립 담당자가 조립 파일에서 다시 만드는 방식이에요.
시간이 걸리지만, 태블로 퍼블릭에서 유일하게 확실한 방법이에요.

```
[사전 준비]
1. 팀원이 시트 완성 후 디스코드에 "스크린샷 + 필드 구성" 전달
   전달할 내용:
     - 시트 스크린샷 (완성 형태)
     - 행·열 선반에 올린 필드 목록
     - 마크 카드 설정 (색상·크기·레이블·세부 정보)
     - 필터 설정
     - 서식 설정 (색상 코드, 폰트 크기 등)

[조립 담당자 작업]
2. 조립 파일(dashboard_v1_YYYYMMDD.twbx)을 열어요
3. 새 시트 탭을 추가해요
4. 팀원이 공유한 필드 구성을 보면서 동일하게 시트를 만들어요
5. 스크린샷과 비교해서 결과가 같은지 확인해요
6. 시트 이름을 합의한 차트 제목으로 변경해요

[반복]
7. 모든 팀원 시트에 대해 2~6 과정을 반복해요
```

> **팁**: 시트가 많으면 시간이 오래 걸려요.
> 그래서 **방법 A(화면 공유)를 우선 권장** 해요.
> 방법 A로 진행하면 처음부터 한 파일에서 만들기 때문에 취합이 필요 없어요.

### 취합을 줄이는 현실적인 전략

```
방법 A+B 혼합:
  → 조립 파일을 하나 만들고, 화면 공유 상태에서 시트를 하나씩 같이 만들어요
  → 시트 1개당 조작 담당자를 교체하면 모든 팀원이 경험을 쌓을 수 있어요
  → 이 방식이면 취합 과정 자체가 필요 없어요
```

### 취합 전 체크사항

- [ ] 모든 팀원이 **동일한 clean 데이터 파일** 을 사용하고 있는가?
- [ ] 각 시트 제목이 인사이트를 담고 있는가? ("이용객 차트" 아닌 "2호선이 이용객 1위")
- [ ] 색상·폰트 규칙이 모든 시트에서 통일되어 있는가?
